# 1.0.6 功能升级与首次更新引导设计

## 目标

交付一个可信、可验证的 `1.0.6` 版本：用户只能操作已具备完整链路的协议能力；可以组织和快速定位主机；可以在协议能力允许时传输文件和同步纯文本剪贴板；并在升级后的首次启动看到本版本实际交付的更新。

## 范围

`1.0.6` 包含以下功能包，按依赖顺序交付：

1. ArkTS/native 测试验收基线与 VNC/协议能力真值化。
2. 主机工作台：收藏、分组、排序、最近连接、健康与延迟展示。
3. 可靠内容交换：SFTP 任务状态、RDP 共享盘可用性、RustDesk 文件传输状态。
4. RDP 与 RustDesk 的双向纯文本剪贴板。
5. 每版本首次启动的更新 Swiper，版本号提升到 `1.0.6`。

不包含 VNC/RFB 实现、麦克风上行、多显示器、录制、聊天、隧道、摄像头或企业设备重定向。它们必须分别设计、实现和验证后才可进入后续版本。

## 启动与更新引导

### 入口与优先级

`EntryAbility` 继续加载 `pages/GuidePage`。`GuidePage` 在读取 `RemoteDesktopAppPrefs` 后按照下面的优先级决定页面模式：

1. `guide_shown_v4` 不存在或为 `false`：显示既有首次使用教程。
2. 教程已完成、当前版本有已发布的更新条目、且 `last_seen_release_version` 小于当前版本：显示当前版本更新 Swiper。
3. 其他情况：直接跳转 `pages/LoginPage`。

全新安装只显示首次使用教程。首次教程完成时，除写入 `guide_shown_v4=true` 外，同时把 `last_seen_release_version` 写为当前版本；因此新用户不会在下一次启动时被连续弹出更新页。

升级用户必须在更新 Swiper 最后一页点击“继续”后，才写入 `last_seen_release_version`。中途退出、崩溃或杀进程不写入状态，下次启动继续展示该版本更新。未知版本、降级版本或空更新条目不展示更新页。

### UI 规则

更新 Swiper 直接沿用 `GuidePage` 的主题、沉浸式安全区、图标卡片、分页点、动效与布局比例，不使用新的视觉体系。更新模式的底部操作仅有“上一步”“下一步”和末页“继续”，不提供跳过或关闭入口。

每一个已发布版本在更新注册表中定义 2–3 个真实功能重点和固定的欢迎末页。`1.0.6` 的最终页固定为“欢迎使用 1.0.6”，按钮为“继续”。前三页必须由本次已验收的能力生成：任何未完成、未通过测试或未获得设备证据的功能都不得出现。

### 架构边界

- `ReleaseNotesPolicy.ets`：纯版本比较、首次展示判定、模式选择与标记写入条件；不导入 ArkUI 或 Preferences。
- `ReleaseNotesRegistry.ets`：版本到更新页面数据的静态注册表；仅登记已发布版本。
- `GuidePage.ets`：读取/写入 Preferences、调用策略、复用现有 Swiper 外观并路由到登录页。
- `ReleaseNotesPolicy.test.ets`：覆盖新装、升级、已读、中断、未知版本与降级分支。

将版本号从 `1.0.5` / `1000005` 升至 `1.0.6` / `1000006` 的操作必须在所有 `1.0.6` 功能完成、更新注册表最终内容确定时进行，避免测试包或中间 HAP 误触发用户更新提示。

## 功能设计

### 1. 可验收基线与能力真值

`HostListPage.ets:2627` 的 ArkTS 语法错误必须先被隔离和修复，随后重新运行 `default@OhosTestBuildArkTS`。现有扩展注册表测试要纳入主 native 测试目标，使协议注册和能力声明可重复验证。

新增 `RemoteSessionCapabilityPolicy.ets` 作为唯一的用户能力判定源。它根据协议、连接状态、RDP 共享盘挂载状态、SFTP 状态、RustDesk 已验证 peer 与入站剪贴板回调等运行时事实，返回每个动作的 `enabled` 与用户可读的原因。未知协议和缺失事实默认禁用。

顶栏、文档选择、拖放、剪贴板监听和成功提示都使用该策略。VNC 在 `vnc_adapter.cpp` 仍是 mock 时必须返回明确“不支持”，不允许进入伪连接会话。RDP 共享盘只在 native 报告挂载成功后可操作；RustDesk 未获得每传输完成确认或入站剪贴板回调前不得声称完整可用。

### 2. 主机工作台

`RemoteHost` 与 `CloudStore` 已保存 `isFavorite`、`groupId`、`sortOrder`、`lastConnected`、`lastHealth` 和 `lastLatency`。本版本复用这些字段，不新增云表、不进行凭据共享，也不承诺跨设备同步结果。

增加纯 `HostWorkspacePolicy`，实现分组规范化、分组列表、收藏优先、最近连接、用户排序与连接健康文本。搜索范围扩展到标签、地址、用户名、协议和分组；密码、私钥、Token、TOTP 永远不能参与搜索、排序或日志。

主机页在不改变原有证书、SSH 前置检查、锁定和连接路由的前提下，增加收藏切换、分组分配、排序选择、最近筛选与健康信息。每次修改由 `HostSyncService` 先持久化，再更新页面数据源；持久化失败时页面状态保持原值并显示可追溯错误码。

### 3. 可靠内容交换

所有传输在 ArkTS 侧统一为 session-scoped `TransferSession`，包含唯一 id、协议、方向、文件名、总字节、已传输字节、阶段、取消能力和诊断码。合法阶段为 `preparing`、`transferring`、`waitingRemote`、`completed`、`failed` 与 `cancelled`。终止状态不能回到传输中。

SSH/SFTP 是首个迁移对象：保留现有 512 MiB 上限和 64 KiB 分块，增加取消、失败重试与断点续传语义。取消必须停止后续分块、关闭本地句柄、保留远端不完整文件状态，并禁止成功 toast。设备验证覆盖上传、下载、取消、断网和恢复后重试。

RDP 共享盘保持两阶段异步挂载：连接成功不等于共享盘可用。native 需要向 ArkTS 暴露挂载成功/失败状态和诊断码；用户只能在状态为已挂载时向共享盘写入。Windows 端 `\\tsclient\\<drive>` 可见、读写和挂载失败不影响桌面连接是启用该能力的设备门槛。

RustDesk 传输不能继续一次性把整个文件读入内存或用全局 `LAST_ERROR` 文本确认完成。它改为受大小上限保护的分块/流式发送、每任务类型化进度、取消与完成确认。下载能力仅在协议端具有专属的文件接收、路径选择和完成回调后开放；否则必须在能力策略中禁用。

### 4. 双向纯文本剪贴板

本版本仅支持 UTF-8 纯文本。`ClipboardSyncPolicy` 负责方向开关、最大长度、来源令牌和回环抑制；二进制、文件、超长文本和无法验证来源的内容一律拒绝，日志不可记录原文。

RDP 需要基于实际 FreeRDP `cliprdr` 协商通道实现发送与接收回调，而不是只写入 `impl_->clipboardText`。RustDesk 需要从 Rust `connector.rs` 解析入站剪贴板消息，经 FFI、C++ bridge 与 NAPI 派发给 ArkTS。每个协议的接收能力只有在活动会话回调已安装并完成原生 fixture/策略测试后才可开启。

设备验证至少覆盖 RDP 与 RustDesk 的本地到远端、远端到本地、收发任一方向关闭、超长文本、Pasteboard 不可用、网络重连和双端回环。任何失败必须保留原有本地剪贴板并给出不含原文的诊断提示。

## 验收与发布

每个功能包完成前都必须有对应的纯 ArkTS 测试；原生/FFI 新增分支还必须有 native 或 Rust 测试。生产 `assembleHap`、ArkTS 测试目标、注册后的 native 测试和针对 Rust 改动的可重复 Rust 测试要记录在同一提交的验证报告中。

设备验收包含成功路径与失败路径：主机工作台持久化、SSH 上传/下载/取消、RDP 共享盘挂载/远端可见/失败、RustDesk 上传/取消/下载（如启用）、两协议剪贴板双向与回环、升级首次展示与同版本不重复展示。更新 Swiper 的每一页内容必须能链接到对应功能的测试和设备证据。

发布前仍需满足项目的安全硬门槛：敏感云字段不得明文回退，账号数据必须隔离，令牌安全存储，RustDesk 依赖来源和许可证可复现。任一硬门槛未完成时可保留开发构建，但不能作为稳定发布候选。

## 后续版本

`1.0.7` 优先产品化后台会话连续性和 PC/无障碍/i18n 基础。每个后续版本都在功能验收完成后新增 `ReleaseNotesRegistry` 条目，并沿用本设计的首次展示和证据要求。
