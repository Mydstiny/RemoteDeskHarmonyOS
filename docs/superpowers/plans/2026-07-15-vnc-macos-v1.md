# VNC macOS V1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不改变 RDP、RustDesk、SSH/SFTP 行为的前提下，用标准 RFB 和独立 VNC 密码成功连接 macOS 内置屏幕共享，稳定显示桌面并完成键鼠控制。

**Architecture:** `VncAdapter` 只管理产品生命周期；项目自有 `IVncEngine` 隔离 LibVNCClient，`VncFramebuffer` 和 `VncFramePump` 生成有稳定生命周期的 BGRA 帧，再通过独立 raw-pixel callback 进入现有 generation-safe renderer。所有功能由握手后的 `VncCapabilities` 决定，第三方类型不得越过 engine 边界。

**Tech Stack:** HarmonyOS NEXT API 23、ArkTS、C++17、LibVNCClient、CMake、现有 GLRenderer/RendererNapi、项目轻量 native test runner。

## Global Constraints

- 只在唯一工作区 `C:\Users\14288\DevEcoStudioProjects\RemoteDesktop` 执行；当前 RDP 分支闭环前不开始 VNC 功能修改。
- VNC 必须在新的独立 `codex/vnc-macos-v1` 分支实施；不得把本计划混进 `codex/rdp-file-clipboard` 的功能提交。
- VNC 构建或运行失败时保持 feature gate 关闭，不得影响 RDP、RustDesk、SSH/SFTP 的注册、连接、渲染和断开。
- V1 只支持直连 TCP、RFB 3.3/3.7/3.8、独立 VNC 密码、Raw/CopyRect、BGRA 显示、基础键鼠、滚轮、view-only 和客户端缩放。
- 默认端口为 TCP 5900；`None` authentication、Apple High Performance、macOS 账户认证、音频、文件传输、Repeater 和动态远端分辨率均不得在 V1 宣称可用。
- 最大边长 8192、最大总像素 33,554,432、最大 framebuffer 134,217,728 bytes、最大服务器名称 4096 bytes；所有乘法和矩形运算必须检查溢出。
- connect timeout 10 秒、认证 timeout 15 秒、首帧 timeout 15 秒、message-loop poll 50 毫秒、disconnect join 上限 3 秒。
- LibVNCClient 必须固定到包含当前已知 client decoder 安全修复的明确 commit；Tight、ZRLE、Hextile、JPEG、TLS、SASL 和 server 组件在 V1 保持关闭。
- 依赖变更同步更新 NOTICE、SBOM、provenance、双 ABI artifact hash 和许可证；不得提交本机 SDK 路径或签名文件。
- 所有实现任务先写失败测试，再写最小实现；每个任务只暂存列出的文件并独立提交。

```cpp

    virtual void close() = 0;
```

## 后续升级顺序（不进入 V1 产品开关）

1. **V1.5 macOS 账户认证：** 先抓取目标 Mac advertised ARD auth types，单独做协议/许可证/凭据威胁模型；仅在有规范和真机矩阵后增加 `MACOS_ACCOUNT` engine security provider。
2. **V2 互通与交互：** 每个 encoding 以独立 decoder、安全上限、fuzz corpus 和服务端矩阵加入；先 Hextile/ZRLE，再审计后的 Tight/JPEG；随后文本剪贴板、remote cursor、continuous updates 和完整 IME。
3. **V3 安全 transport：** VeNCrypt/TLS/X.509、证书确认/pinning、SSH tunnel、SOCKS/HTTP proxy；公网 VNC 默认要求加密 transport。
4. **V4 网关与桌面扩展：** ExtendedDesktopSize、公开多屏扩展、UltraVNC Repeater、reverse/listen；新增独立 typed gateway model，不复用 RustDesk relay 凭据模型。
5. **V5 厂商扩展：** 文件传输等能力按厂商协议分别实现和声明。Apple High Performance 保持独立研究 engine，不阻塞也不污染标准 RFB 基线。

每一阶段都必须继续复用 `IVncEngine`、transport、安全 provider、encoding registry、framebuffer pump 和 capability truth；如果某项能力要求绕过这些边界，应先修改设计并单独评审，而不是在 adapter/UI 中添加协议特例。
